# 2026-07-09 Girl-Agent 真实感改造

## 做了什么

对 Girl-Agent 项目进行了全面的"真实感"改造，解决三个核心问题：AI 感太重、缺少"要不要回"判断、回复时机不够真人。

### 改动清单

1. **Prompt 改造** (`prompts.py`)
   - 新增 `state_to_hint()` 函数，把 15+ 个数值字段翻译成自然语言行为提示
   - 从"好奇心: 73/100"变成"你对他说的事有点好奇"
   - 精简禁令从 5 条"不要"压缩成 1 条正向原则
   - `conversation.py` 温度从 0.85 降到 0.75
   - `PromptedConversationCore` 新增 `example_messages` 参数注入 few-shot

2. **Sanitizer 扩充** (`sanitize.py`)
   - 从 4 个模式扩充到 22 个 AI 腔模式（"说实话/讲真/其实吧/好问题/有趣的是"等）
   - 改用列表循环替代逐条正则

3. **消息合并策略** (`turn_taking.py`)
   - 新增 `long_burst_seconds=15.0` 参数
   - pending_count>=3 且 continuation 时用长等待
   - pending_count>=5 时强制长等待

4. **ReplyDecision 层** (`reply_decision.py` - 新文件)
   - 消息分类: urgent/question/ack/emotional/story/statement
   - 忙闲判断: 基于 human_rhythm 时段 + 随机概率
   - 决策: reply_now / defer(10-45分钟) / skip
   - "在吗"打断检测

5. **延迟回复机制** (`qq_websocket.py`)
   - `QQMessageCoalescer` 新增 `_deferred` 和 `_deferred_tasks` 字典
   - defer 时存入 pending，设计时器
   - 计时器到期或被新消息打断时触发回复
   - 延迟回复带 `DEFERRED_CONTEXT_HINT` 提示"刚在忙"
   - `enable_reply_decision` 参数控制是否启用（默认 False，生产环境 True）

6. **Engine 改造** (`engine.py`)
   - `handle_message` 新增 `skip_reply` 和 `context_hint` 参数
   - skip_reply=True 时处理状态但不调用 LLM
   - `has_unread` 字段随 skip/reply 自动更新

7. **MoodState 扩展** (`models.py` + `db.py`)
   - 新增 `has_unread: bool = False` 字段
   - 数据库迁移: `_ensure_column` 自动添加列

### 第二轮：进一步降低 AI 味

用户反馈"还是有点 ai 味重"，追加了以下改动：

- `prompts.py`: 从 reply_prompt 移除 `emotion_context_line`（仍在输出"情绪向量: 信任 30%"百分比），`state_to_hint()` 已覆盖
- `prompts.py`: 回复原则改为全正向："回复尽量短，一两句就好""你不是助手，不用帮他解决问题"
- `human_rhythm.py`: 移除"不要写舞台动作、括号动作或系统说明"，改为"你的消息就是纯粹的私聊文字，像微信打字一样"
- `character.yaml` base_prompt: 6 条"不要"指令改为正向表述
- `character.yaml` speech: 移除"不装腔/不写小说段落/不会写旁白"
- `character.yaml` style_rules: 4 条"不要"改为正向
- `character.yaml` boundaries: "不要"改为正向
- 更新 test_engine.py, test_human_rhythm.py, test_character.py 断言

核心思路：LLM 的 attention 机制会让"不要做 X"中的 X 被激活。全改成正向指令后，模型不再被负向词干扰。

### 第三轮：few-shot 扩充 + 两遍生成

- `character.yaml` example_messages 从 5 条扩到 12 条，覆盖：被问作息、分享天气、室友吵架、帮写代码、在吗、心情询问、读书分享等场景
- 新增 `PromptedConversationCore._rewrite_for_naturalness()` 两遍生成：flash 出初稿 → 短 prompt 让模型判断"像不像真人" → 改写
- rewrite prompt 极简："如果这句话听起来像AI/客服/助手说的，改写得更自然。够自然就原样返回。只输出消息。"
- `config.py` 新增 `ENABLE_REPLY_REWRITE`(默认 False) 和 `DEEPSEEK_REPLY_MODEL`(可选，默认用同模型)
- `runtime.py` 根据配置创建 rewrite_model 传入 engine
- `engine.py` 新增 `rewrite_model` 参数，传给 fallback PromptedConversationCore
- 用户启用方式：`.env` 里加 `ENABLE_REPLY_REWRITE=true`，可选 `DEEPSEEK_REPLY_MODEL=deepseek-reasoner`

### 第四轮：Codex 接手改进

用户用 Codex 做了 5 个 commit（3fb38fb..48b905c），+1931 行，40 个文件，152 测试全过。

**关键新模块：**

1. **`process_lock.py`** (59行) — `SingleInstanceLock` 文件锁，防止多个 `companion-qq-ws` 进程同时运行。**这才是消息重复的真正根因**——不是 QQ 投递两遍，是两个进程都在收 WebSocket 事件。

2. **`dialogue_eval.py`** (309行) — 完整的对话自然度评估框架：
   - 15+ 种问题检测：stage_direction/acquaintance_crutch/assistantese/problem_solver/flattened_question/ungrounded_local_detail/ungrounded_self_event/stereotype_reply/unsupported_memory_claim/question_nag/emotion_question_only/incomplete_trailing/low_engagement/echo_only/too_many_questions/too_after_question/too_long/too_many_lines
   - 4 个测试场景：ack处理/地点回答/长故事/情绪消息
   - 可运行：`uv run companion-dialogue-eval --live`

3. **`reply_postprocess.py`** (132行) — 回复后处理规则：
   - 移除连续追问（如果最近已经问过）
   - 修复不完整结尾句（"就是..."→"。"）
   - 情绪消息只回了问句时→替换为共情陈述
   - 回复太薄时→补充存在感
   - 空回复+有意义用户消息→补充存在感

**关键改进：**

- `qq_websocket.py`: 去重先查 message_id，None 时才用文本指纹；`main()` 加 `SingleInstanceLock`
- `reply_decision.py`: 新增 `mark_unread` 字段；用 `state.mood` 调整忙闲概率；`has_unread` 触发立即回复
- `conversation.py`: 集成 `postprocess_reply_text()` 在 rewrite 前后各跑一次；SillyTavern 客户端加 `trust_env=False`
- `config.py`: 新增 `enable_reply_decision` (默认 True)
- `engine.py`: `handle_message` 新增 `mark_unread` 参数
- `sanitize.py`: 更多 AI 腔模式
- `prompts.py`: 保留 `state_to_hint()`，增加更多上下文

**Codex 比我做得更好的地方：**
- 发现了进程锁这个根因（我只做了文本去重，治标不治本）
- `dialogue_eval.py` 是系统化的评估框架，可以持续衡量回复质量
- `reply_postprocess.py` 在 LLM 输出后做规则修正，比单纯靠 prompt 引导更可靠
- 去重逻辑更合理：先查 ID，None 时才用文本（避免误杀用户故意重复的消息）
