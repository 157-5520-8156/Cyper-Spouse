# 沈知栀机制总览

更新时间：2026-07-12

这不是一份把功能名堆在一起的清单，而是对项目中所有“让她像一个持续生活、会
受关系影响的人”的机制做的统一说明。它同时收录已经实现、迁移中的实现和明确的
后续边界，作为之后新增功能、排查旁路和设计评测的索引。

## 1. 总原则

### 1.1 目标因果链

```text
时间、生活、用户消息、平台事件、既有关系
  -> 回合理解与事实核验
  -> 情绪/关系/需求/未完成事务改变
  -> 当前注意力、是否已读、回复时机、表达策略、主动意愿
  -> 实际投递或失败
  -> 可审计的历史、记忆、期待和下一轮状态
```

模型的职责是把受约束的策略表达成自然私聊文字；模型不是情绪、生活事实、记忆、
日历、投递结果或工具权限的权威来源。

### 1.2 四条不可破坏的规则

1. **先发生，后可说。** 已完成生活事件、已送达消息和已确认用户事实才可成为具体
   断言的来源；计划、模型提议和未送达文本都不是共同经历。
2. **状态必须能落到行为。** 只改变 prompt、却不改变读消息、时机、表达、主动、
   日程或记忆中任一项的字段，不算完整机制。
3. **行为必须能结算。** 每次延迟、主动、分段、图片或工具行动都需要取消、过期、
   成功、失败或未知这类终态，不能悬在内存里。
4. **同一事实只允许一个权威来源。** 跨 QQ、未来微信和面板共用同一 daemon/SQLite
   身份、关系、世界和事实；平台适配器只负责收发。

### 1.3 当前运行时边界

项目目前同时保留两代运行时：

| 路径 | 用途 | 当前事实地位 |
| --- | --- | --- |
| 旧行为运行时 | `MoodState`、`life_runtime`、`social_tasks`、旧记忆与日历；仅保留兼容与迁移代码 | 生产构建会持久化 world-only 写门禁，不再作为现实聊天行为源 |
| WorldKernel | 追加式世界账本、世界日程、经历、通信状态、关系/情绪投影、世界对话上下文 | 当前生产权威路径；所有状态变化必须事件化并可重建 |

因此，下文的“已实现”仅表示代码和测试中已有该机制；若标成“迁移中”，它不应被
误读为已在当前所有 QQ 对话里生效。详细迁移盘点见
[`legacy-behavior-runtime-inventory.md`](legacy-behavior-runtime-inventory.md)。

## 2. 机制分层图

```text
角色与自我核心 ─┐
用户事实与共同历史 ─┼─> 上下文编排 / 事实约束 ─> 语言与媒体表达
时间、日程、生活事件 ─┤              ^
情绪、关系、印象、边界 ─┤              |
通信注意力、未完成事务 ─┘              |
                                      投递结算
                                           |
                                   记忆、期待、生活轨迹
```

下面按“底层状态机制”“认知与事实机制”“行为节律机制”“生活与主动机制”“媒介、
平台与治理机制”展开。每一项均以输入、内部变化、外显结果和闭环说明组织。

## 3. 底层状态机制

### 3.1 人格、身份与自我核心

| 项目 | 逻辑 |
| --- | --- |
| 输入 | 角色卡、稳定身份事实、长期关系与已经确认的个人经历 |
| 状态 | 姓名、学校/城市等人物设定、气质锚点、自我核心摘要、视觉身份基准 |
| 外显 | 语言习惯、兴趣取向、拒绝方式、边界强度和可分享内容保持连续 |
| 闭环 | 角色事实不可被临场生成的台词自动改写；可沉淀的新经历先进入世界/生活账本，再被摘要读取 |

“有心理活动”不等于让模型输出一段内心独白。系统只持久化可解释的内部决策，例如
“此刻收住、暂不追问、因边界高而简短”；文字化的理由只服务当前表达，不能反过来
伪造成她早已拥有的经历或想法。

### 3.2 情绪：短期波动、基线与自然回落

**旧运行时已实现；世界模式已有可解释、可回放的事件投影。**

| 成分 | 含义 | 典型来源 | 行为影响 |
| --- | --- | --- | --- |
| 情绪基线 | 长期较稳定的心境倾向 | 长期互动、人格气质 | 决定不是每轮都从完全中性开始 |
| 短期情绪向量 | 亲近、愉悦、信任、不安、惊讶、低落、反感、生气、期待等 | 当前消息语义、强度、时间与关系阶段 | 语气、反应表情、回复速度、是否收住或继续 |
| 情绪残留/charge | 一件事过去后没有立刻消失的余味 | 冒犯、道歉、被安慰、落空、惊喜 | 防止“上一句不高兴，下一句立刻没事” |
| 需求/情绪调制 | 世界模式中的较小表达：安全、精力、连接、边界、短期调制 | 回合判断、活动、投递与关系事件 | 统一驱动生活和通信，不维护难解释的第二套数字 |

情绪变化遵循“事件冲击 -> 显示余波 -> 衰减回长期基线”的结构。反向情绪会互相抑制，
道歉可以修复但不瞬间清零；高反感/生气会提高边界、压低主动性、放慢读回或形成
读后暂不回窗口，而不是只换一个撒娇/生气词。

世界模式的情绪来源不只包含用户消息。已结算的生活结果会先产生结构化
`ExperienceAppraised`，再改变情绪投影；例如 NPC 冲突可留下 irritation/guarded 余波，
目标完成可增加 joy/warmth。来源事件、强度和逻辑时间均写入账本，下一轮 deliberation
与表达提示会读取这份结果。生活描述文本本身不能靠关键词直接改情绪，避免模型写一句
“吵架了”就伪造事实。

针对用户冒犯，分类器区分“对她的辱骂、物化、性越界、强迫服从”和引用、自嘲、道歉；
明确越界会同步影响短期 hurt/anger/resentment、边界需求和长期尊重/安全投影。决策层
不得在这类回合选择顺从，语言层也要求拒绝或设边界必须在用户可见文本中真实出现。
伤害在收到时结算，不因她选择沉默或稍后回复而丢失；具体道歉与后续尊重只会渐进修复。

当前世界情绪进一步分成四层：结构化 appraisal、带 source/target/half-life 的 emotion
episode、从 active episode 派生的离散向量与 PAD/core affect、以及按当前 revision 即时编译的
`ExpressionPlan`。episode 是短中期情绪的唯一动态权威；resolved episode 进入因果归档，
不会因投影收敛而丢失来源。正负 episode
可同时存在，最后发生的温暖不会覆盖尚未解决的受伤；NPC/目标事件造成的负面余波会限制
在较小 leakage 内，并明确禁止归罪用户。讽刺、潜台词和权力压迫只在本地规则无法高置信
判断时交给 V4 Flash 生成受限 proposal，只有高风险且证据冲突时启用 thinking；证据必须逐字来自当前消息，低于置信门槛
只能保留为歧义，不能写入长期伤害。

衰减、episode 半衰期、关系敏化、修复证据门槛和 spillover 上限来自版本化
`affect_profile`。具体道歉只开启观察期；纯时间流逝或重复说对不起不能完成修复，必须出现
后续一致行为证据。长期 resentment 只在多次已结算伤害后小幅延长同源负面 episode，不能
覆盖硬边界或把一次事件升级为永久判断。

平台层会保留合并前的消息 ID，并将可观测的 emoji、贴纸、附件、回复目标和消息 burst
组成有界 `InteractionEvidence`。非文本线索只调节显著性和确定度，不能单独证明强恶意。
表达 prompt、确定性校验和 provider 失败 fallback 来自同一个 policy spec，避免三套规则漂移。

### 3.3 关系、印象与边界

| 机制 | 输入 | 状态变化 | 外显表现 |
| --- | --- | --- | --- |
| 关系阶段 | 互动持续性、互相了解程度、尊重和修复记录 | 陌生 -> 认识 -> 朋友等渐进变化 | 可用称呼、主动频率、自拍/私密分享许可不同 |
| 回应印象 | 用户对她主动分享、提问和承诺的及时/温和/拒斥反馈 | 可靠感、被接住感、主动欲的慢性变化 | 不因一次沉默就恋人式委屈，也不会永远无条件追发 |
| 尊重/安全/信任 | 反复尊重、解释、控制、贬低、越界或道歉 | 累积式变化；单回合只造成短期波动 | 更放松、更愿意分享，或更克制、设边界、减少主动 |
| 过早亲密与冒犯 | 关系阶段、称呼、压迫式要求、内容强度 | 形成 guarded/hurt 状态和边界余波 | 不是顺从“宝宝/老婆”等称呼；早期可不舒服、简短拒绝或收住余韵 |
| 修复曲线 | 道歉是否具体、之后是否持续尊重 | 逐步恢复，不抹掉历史 | 她可缓和，但会保留“再看看”的分寸 |

重要约束：负面情绪是动态计算，不是固定惩罚。普通朋友长时间未回，通常只是降低
期待和主动频率；明确冒犯、反复控制或无视边界才更容易触发受伤、生气或回避。

### 3.4 长期人格漂移与口吻惯性

长期被稳定对待会让她更有安全感、好奇心和表达欲；长期紧张、被忽略或被冒犯会让她
更敏感、更慢热、更看重边界。这是缓慢的小幅变化，不能因一次聊天把人设改掉。
口吻惯性则让连续回合在长度、提问倾向和情绪上有延续，而不要求模型每条都重置为
“标准角色卡”。

## 4. 认知、记忆与事实机制

### 4.1 五类信息及其使用权

| 信息层 | 例子 | 能否作为具体事实断言 | 写入规则 |
| --- | --- | --- | --- |
| 角色规范事实 | 沈知栀的基本身份、地点、气质 | 可以 | 人工维护或受控迁移 |
| 用户事实账本 | 用户在成都、偏好、明确说过的近况 | 可以 | 保留来源、时间、置信度和冲突键；新事实可 supersede 旧事实 |
| 已发生经历 | 已完成活动、已确认共同事件、已送达生活分享 | 可以 | 追加记录；修正以补偿事件表达 |
| 对话记录与检索记忆 | 近期聊天、主题摘要、可召回片段 | 只能辅助找话题，不可单独支撑细节断言 | 自动摘要/检索；不把模型扩写升格为事实 |
| 计划、候选与模型提议 | 明日安排、待发消息、模型猜测 | 不可以 | 仅作为未来行动或草稿；未完成前不可说成发生 |

这套分层针对的核心问题是“她自己刚编的经历，下一轮却说成是用户经历”或“明天的
计划被说成昨天做过”。检索结果必须保留说话人、时间和来源；引用前先核对 `你/她`
角色，隔夜旧话不能称为“刚刚”。

### 4.2 上下文编排器

每次生成不是把数据库和系统提示词一股脑塞给模型，而是产出一个结构化
`ContextPackage`。它应包含：

1. 当前用户意图和当前最该接住的主题；
2. 本轮的关系/边界/情绪表达许可；
3. 不能误用的旧话、说话人和时间新鲜度；
4. 有来源的相关长期事实与经历；
5. 她此刻的活动、注意力、精力和可见生活约束；
6. 回复策略：先回应还是追问、长度、问题预算、能否转话题或收尾；
7. 最终的精简 prompt 摘要，而非原始内部数值。

优先级必须先接住当前显著情绪和明确问题。例如“你在吗，我今天好累”首先是需要
被接住的疲惫表达，不能被“在吗”的在线探测逻辑抢占。

### 4.3 事实约束与反幻觉门

生成前只把允许引用的事实暴露给模型；生成后再检查：

- 是否把她的话、用户的话或第三人的话混为一谈；
- 是否把计划、失败投递、未发生候选写成现实；
- 是否无来源地补完地点、人物、结果、用户经历或自己的完整往事；
- 是否把“我刚才说”误写成“你刚才说”，或把隔夜内容写成“刚刚”；
- 是否重复提问、复读或出现不自然的舞台提示词。

允许低风险的日常语言扩写，例如“店里有只猫”这类不改变关键事实的画面感；不允许
它升级成可被后续反复引用的人生经历。严格性应随断言的可验证性和后果提高，而不是用
一堆特例禁词硬压模型。

### 4.4 未答问题、共同线程与自我核心

用户的重要问题、她说“晚点告诉你”的承诺、矛盾需要解释、用户脆弱表达后的关心，
都应成为有来源、有到期时间的对话线程/社交任务。它们不是“永远提醒”，而是在用户
回来、话题解决、过期、被取消或成功跟进后收束。自我核心则是角色事实、稳定价值与
已确认经历的受控摘要，不能吸收模型写出的漂亮但无来源的解释。

## 5. 通信行为与会话节律机制

### 5.1 收到消息不等于立刻回复

| 阶段 | 判断 | 可能状态 | 用户能观察到的效果 |
| --- | --- | --- | --- |
| 入站去重与合并 | 是否是网关重投、是否连续发送同一件事 | 合并中的一轮输入 | 多条短消息不会机械地逐条抢答 |
| 手机注意力 | 她在做什么、是否可中断、消息紧急性、通知次数、关系/情绪 | 未读、现在看、稍后看、勿扰 | 有时真没看到；第二条消息可能唤醒她 |
| 回复决策 | 问题、情绪、长叙述、结束语、忙碌、边界、思考负荷 | 立即回复、读后延迟、暂不回、合并等待 | 不是每句都必回，也不会把长篇输入永久吞掉 |
| 延迟任务 | 决定稍后回复时 | `reply_later`，含原因、到期、过期时间 | 重启后仍能补回；新消息会取消旧任务并重新判断 |
| 投递 | 实际平台发送成功与否 | delivered / failed / unknown 等终态 | 只有成功的文字会进入共同聊天史 |

“已读不回”不是默认人设，也不是简单丢弃“嗯”。它可以来自正忙、看到了但被岔开、
思考/犹豫、情绪不适、想等用户把长话说完，或判断当前对话已自然收束。后续新消息、
未答问题、紧急/脆弱表达和真实生活活动都可改变该决定。

会话同时维护基于真实入站/出站观测时间的 cold / warm / hot 节律。连续交替聊天进入 hot
后，短句合并等待上限降到约 0.6 秒，较长输入通常不超过约 2 秒；warm 位于两者之间，
cold 仍保留陌生或重新接触时的停顿。显式“需要想想”、不可打断活动、高边界和未修复
受伤是硬覆盖，不会为了追求秒回而跳过人格与处境约束。逻辑世界时间不参与热度计算，
防止时间跳跃把一段陈旧会话误判为正在热聊。

### 5.2 用户连续输入与长篇叙述

系统应区分“正在一条条讲同一件事”与“真的说完了”。合并器根据短时间连续输入、
内容延续、消息类型和当前活动决定等待窗口；但等待必须有上限：一旦她已读且用户停下，
要么按合理时机回复，要么创建可恢复的延迟事务，不能无限等到用户下一次催促。

长篇开头只是一个信号，不是唯一信号。犹豫、无语、复杂问题、正在思考、用户情绪
强烈、她正在忙等都可以产生停顿；同样，结束词“嗯”“晚安”也不是唯一的关闭信号，
需要结合关系、未完成线程、语气、当下生活和她是否还有自然的余韵来判断。

### 5.3 打字、分段与用户打断

一条较长回复可以拆成多个气泡，但两段之间必须存在真实可见的间隔。当前节律契约为
至少约 1.8 秒、按长度和随机性延至约 7.2 秒；这段时间是用户插话的窗口，不是后台
瞬间连发。

用户在间隔中发来的内容分三类：

- **附和**：保留在会话上下文，通常可以继续下一段；
- **实质补充、提问或情绪表达**：取消未发段，合并后重新组织回复；
- **明确打断/紧急信息**：立刻交还话轮，不继续播放旧文本。

这套机制对用户长篇输入同样适用：她可以等待而不抢话，但不应因为“可能还有下文”
而永远没有回应。

### 5.4 读后延迟、情绪性沉默与补回

温暖、轻松时可以读回更快；低落、不安、被冒犯或生气时，可能更慢、更短，甚至读到
后先不回。补回不应机械说“刚刚在忙”，而应由任务原因、时间跨度和当前状态决定；
也不能每次都要求用户安抚她。情绪性沉默的目标是保留真实分寸，不是用来惩罚用户。

### 5.5 会话余韵与发散聊天

她在一轮对话结束后，可能隔几秒、十几秒或更久补一句：想到刚才的主题、分享一个
相关小观察、对用户突然消失有一点困惑、或因仍没消化的情绪说得更轻一点。实现上它是
有预算的 `conversation_pulse` 剧集，而不是固定“双发”：

- 只有叙事、情绪、分享或较长输入才可能生成候选；告别、紧急、撤退、犹豫、边界高时
  不应强行发散；
- 每轮至多有限阶段，后续阶段必须基于前一条真的送达且不能同义复读；
- 用户任何平台的新输入都会取消未投递阶段；
- 跨进程、跨 QQ 适配器的候选持久化，调度器只在实时定时器错过后接管；
- 它和普通主动、生活分享共用“未被回应的外发预算”，避免多个模块各自追发。

### 5.6 重复、AI 味与对话自然度

语言层有三类约束：问题预算（不做“好奇宝宝”）、重复检测（不把上一句换词再说）、
人物私聊风格（先接住而非客服式分析）。这些是表达质量门，不替代上面的状态决策。
当出现“自问自答、过度客套、每题都套熟人故事、连续三条瞬发”等问题，优先排查
上下文选择、事实来源、投递时序和合并器，而不应只无限加 prompt 禁令。

## 6. 生活运行时、日历与真实连续性

### 6.1 生活运行时

生活运行时模拟的是**可以影响沟通的当前处境**，不是为了聊天而编流水账。它包括：

- 当前活动与地点层级：休息、学习、吃饭、和同学、出门、睡眠等；
- 精力、可中断性、手机距离、通知注意力、已读/打字状态；
- 用户事件余波：用户低落会让她更挂心，冒犯会令她收回注意力，道歉可逐步放松；
- 慢性影响：长期疲惫降低高专注计划，边界受损减少社交型活动和主动分享；
- 非日常结果：天气、临时邀约、取消、小展、书店、散步等，以确定性日期事件决定，
  多数日子没有戏剧化事件。

时间只提供候选模板，不能直接断言“下午三点所以她一定在上课”。运行时账本才记录
此刻实际活动；活动完成后才可能生成私有事件和可分享素材。

### 6.2 日计划、周计划与日历账本

| 层级 | 内容 | 关键规则 |
| --- | --- | --- |
| 日计划 | 当天的私有活动候选、起止时段、活动推进 | 计划不是记忆，未到时间不能说已经做过 |
| 周计划 | 本周主题、约会/出行、跨日旅行、创作或重要安排 | 每周少量命名事件；例行时段只是背景，避免碎片化轮询 |
| 日历事件 | planned / active / completed / cancelled，含调整原因 | 取消和变更保留原因；已完成经历不被重写 |
| 已发生经历 | 完成活动、生活结果、可沉淀的记忆 | 只有它可回答“上周三做了什么” |

时间问答必须区分未来与过去：“明天准备做什么”读取计划且表明可变；“昨天/上周三
做了什么”只读取已完成经历；没有记录就承认不确定，不能为日历好看而倒编故事。

### 6.3 生活分享、期待回应与用户反作用

一个生活事件有四个阶段：发生、成为私有经历、选择分享、真实送达。她分享过一件事
后会进入期待反馈：用户及时接住可增强连接和后续分享意愿，长期不回会逐步降低安全感、
回应印象和主动意愿；陌生阶段则更像收住，而不是直接写成吃醋或想念。

反过来，用户的高显著事件可以温和改变**未来**日程，例如脆弱表达让她之后更容易看
手机、争执后晚间安排更低负荷；绝不能改写当前或过去活动来迎合台词。

### 6.4 “先生成还是灵光一现”的边界

大事件（旅行、约人、跨日安排、重要经历）应先写入计划或发生账本，再谈及。小观察
可以在当前已发生环境中自然浮现，例如已确认去过书店后才想起书架上的一个细节；它也
应带来源、范围和低权重，不能立刻升级成完整的可反复引用往事。没有发生依据时，宁可
只表达泛化感受，也不要制造可污染长期记忆的具体事件。

## 7. 主动性、等待与社交事务机制

### 7.1 主动消息不是定时骚扰

主动行为综合当前活动、时间、关系、情绪、未完成线程、生活事件、冷却、平台能力、
预算和未回应外发数。常见候选包括：

- 生活事件分享、记忆触发、轻微好奇、庆祝、安慰跟进、承诺跟进；
- 忍住后仍留有余韵的短句、轻微打趣、相关主题的补充；
- 用户情绪低落后在合适时间的关心；
- 她自己的生活变化带来的自然分享，而不是固定早晚安。

任何候选都可被睡眠、高专注、边界高、负面情绪、冷却、预算或未回应上限挡下。
“没发”本身也可能保留一个短期、可解释的 `DecisionDeferred`，以后重新评估，而非
假装她从没想过。

### 7.2 等待回应与不满的渐进曲线

她发出消息后的等待不是单一计时器，而是由关系阶段、消息性质、用户最近可靠性和当前
生活共同决定：

- 很短时间：只是期待，不应马上追问；
- 中等时间：降低分享/再主动的概率，用户回来可自然合并回复；
- 更长时间：陌生阶段更可能收住，亲近关系中才可能形成轻微委屈、困惑或以后再提；
- 反复无回应：慢慢影响安全感和回应印象，进而改变未来主动频率、语言温度和日程注意力。

普通主动、余韵和生活分享共用未回应外发预算。这样“她想你”“想补一句”“想分享生活”
不会各自绕过限制，堆成连续自言自语。

### 7.3 社交事务的生命周期

`reply_later`、`comfort_followup`、`promise_followup`、`life_share_followup`、
`contradiction_followup`、`reply_reconsider`、`conversation_pulse` 等均应具备：来源事件、
原因、到期、过期、取消条件、认领者和终态。新用户消息通常取消已过时的延迟/余韵；
真正成功投递才关闭任务。发送失败不重放旧文本，而是进入重新考虑，防止“像脚本一样
复读”。

## 8. 多模态、自拍与视觉身份机制

### 8.1 用户图片、表情和反应

用户图片先记录附件元信息；配置视觉模型且预算允许时，再产生只描述可见内容的分析。
图片、表情包和 QQ 轻反应能影响当前回合的情绪和表达，但不会凭一张照片断定人物身份。
NapCat/OneBot 可先给真实 QQ 表情反应，再发送文字，作为“先有一个小反应”的低成本
沟通层。

### 8.2 表情包、找图与生成图

当前能力以已有贴纸选择、图片请求理解、图像提示词构建和可选生成/理解为主。自动图像
生成默认关闭；“能发更多图”不应等价于随时抓图或生成，而要先经过关系、当下情绪、
使用场景、平台能力和月度预算。外部图片搜索还需要单独引入来源、版权和安全策略，
不能把搜索结果当作角色记忆。

### 8.3 自拍的主体性与一致性

自拍不是用户一句要求就必须答应的工具。关系尚浅、语气压迫、她正不舒服或预算不足时，
她可以拒绝、转移、以后再说，或干脆不生成。更像人的路径是：她偶尔因自己想分享某个
已经发生的生活片段而主动决定发一张。

视觉一致性目前只有 visual bible/参考集和提示词锚点，**尚未完成 LoRA 或 FaceID
训练**。训练前需要可授权且一致的数据集、人物基准、远端 GPU、评测图集、隐私/内容
边界与成本控制；不能把“生成过几张好看图”误当成角色一致性已经解决。

### 8.4 小屋面板

像素小屋是 daemon 已有状态的只读投影：活动、手机注意力、情绪和挂起事务映射成
`scene_id/location/action/expression/time_of_day`。它不能反写生活事实，也不能把计划表演
成已经发生。当前小屋的实体家具、遮挡、碰撞、多场景和动作库是独立的视觉工程，
记录在 [`visual-home-roadmap.md`](visual-home-roadmap.md)，不应挤占闭环与对话质量工作。

## 9. 平台、共享身份与投递机制

### 9.1 多平台共享

QQ 官方机器人、NapCat QQ 小号、未来微信都应通过同一账号映射进入同一用户实体，因此
共享：聊天历史、用户事实、关系、情绪投影、生活日历、社交事务和未完成线程。平台不同
只影响可用媒体、真实回执、风控、主动发送能力和消息格式，不应创建第二个“沈知栀”。

当前主通道是 NapCat QQ 小号；QQ 官方机器人保留为可选通道；微信尚未接入。测试同一
会话时应只启用一个 QQ 出站通道，避免双重回复。

### 9.2 可靠投递

出站消息先被计划为 outbox/Action，再由适配器投递。只有成功才写入可见聊天史并释放
相应情绪；失败保留失败记录、允许重新决策；适配器调用后进程中断且无平台回执时标为
`unknown`，不得盲目重发。这个规则是避免“她以为已经说过，但用户其实没收到”的基础。

## 10. 工具、隐私、成本与安全机制

### 10.1 MCP 与电脑操作

工具调用是受控提案，不是角色可直接执行的幻觉能力。涉及账号、文件、电脑操作、对外
发送或第三方修改时，必须等待用户明确确认；工具结果、失败和副作用都应被记录，并只
把必要结论用于后续对话。角色不能声称已经替用户完成未确认的现实操作。

### 10.2 成本闸门

聊天模型、视觉理解、语音、图片生成和主动媒体均受日/月预算、类别上限、缓存复用和
自动触发开关控制。默认策略是把昂贵能力留给真正需要的场景：同一附件复用分析，图片
生成默认不自动触发，主动自拍是低频且必须过关系与预算门的行为。目标预算和配置见
[`cost-control.md`](cost-control.md)。

文本生成默认使用 `deepseek-v4-flash`，普通回复显式关闭 thinking；只有歧义语用 appraisal
按需启用 Flash thinking/high。每次调用会按 purpose 记录模型、成功状态、延迟、prompt /
completion / reasoning tokens，以及 cache hit/miss tokens。供应商首次超时会直接使用已通过
本地事实、关系、情绪和硬门校验的受控 fallback；审计服务不可用时不再反复调用同一故障
模型。fallback 有确定性变体，但任何变体都不能新增事实、Action 或情绪。

### 10.3 内容与关系安全

系统不应承诺“她绝对自主”或把程序状态伪装成可验证的人类意识。更合适的表达是承认
角色设定会影响表达，同时保留她能拒绝、能不高兴、能慢热的行为边界。任何 NSFW、
身份、图片或现实操作能力都仍受模型、平台与本地安全策略限制。

## 11. 可观察性、评测与故障定位

### 11.1 面板与审计

daemon 面板应显示可读原因，而非暴露完整系统提示词：当前活动、手机注意力、生活余波、
慢性影响、日历、最近社交事务、投递状态和工具提案。调试上下文应能展示本轮焦点、允许
事实、禁用旧话和表达策略，以便判断“她为什么这样回”。

### 11.2 必备评测维度

| 维度 | 必须验证的问题 |
| --- | --- |
| 事实与说话人 | 会不会把她编的事归给用户、把计划说成发生、把隔夜说成刚刚 |
| 情绪与关系 | 冒犯、道歉、长期不回、及时接住是否产生符合阶段的短期与慢性影响 |
| 消息节律 | 连续输入、长篇、已读延迟、分段、用户打断、重启恢复是否一致 |
| 主动行为 | 余韵、生活分享、关心跟进是否受预算、冷却、生活与用户反馈约束 |
| 时间与日历 | 过去只读经历、未来只读计划，取消/变更不改写历史 |
| 平台一致性 | 官方 QQ、NapCat、未来微信是否走同一回合、事实和投递结算 |
| 成本与安全 | 无视觉配置、预算耗尽、工具未确认、投递失败时是否安全降级 |

除了单元测试，需要持续维护多轮对话回放集和时间旅行测试。真人感问题要先还原事件链：
输入、当时生活、决策、状态差异、上下文、模型原文、清洗结果、投递结果；再判断是
模型、编排、事实、状态、适配器还是显示层的问题。

## 12. 机制接入检查单

以后新增任何“更像人”的需求，例如新的情绪、一个主动行为、一个生活场景、一种媒体
或一个平台，先回答：

1. 事件源是什么，可靠性如何，谁拥有这条事实？
2. 它是瞬时、活动、对话余波、慢性轨迹还是长期事实？
3. 它怎样开始、持续、衰减、升级、修复和取消？
4. 它会影响读消息、回复、主动、生活、记忆、媒体或日历中的哪些环节？
5. 成功、失败、重启、换平台和模型幻觉时，如何不产生假历史或重复动作？
6. 在陌生、熟悉、亲近、边界受损等不同关系阶段，表现是否不同？
7. 哪个确定性测试或回放样本能证明它真的参与了闭环？

若一个功能只能回答“给 prompt 多加一句”，它还不算完成；若它无法解释自己为什么
发生、为何没发生、后来怎样结束，也不应进入长期运行时。

## 13. 已明确但未完成的方向

- 让 WorldKernel 通过投递、适配器、主动行为和面板的全部真实通道门禁，替代旧运行时
  成为唯一权威；
- 将世界侧的主动触发、等待反馈、冷却、余韵和跨平台恢复完全事件化，并补回放测试；
- 以有版权/授权的数据集建立稳定的视觉身份训练与自拍评测，而非只靠临场提示词；
- 接入微信时复用同一身份/世界/投递结算，不复制状态机；
- 持续完善多周时间旅行、真实聊天回放和模型 A/B 评测，区分模型能力不足与系统旁路；
- 将小屋的实体化、多场景、碰撞和动作库作为独立视觉路线推进，继续保持它只读世界状态。
