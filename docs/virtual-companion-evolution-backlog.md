# 虚拟伴侣世界演化：完整优化清单

更新时间：2026-07-11

这份清单的目标不是继续增加“像人”的机制名称，而是按依赖顺序提升三件事：

1. 她说过的、做过的事是否有依据；
2. 她的选择是否能长期保持因果一致；
3. 这些选择是否能逐步被用户感受到。

`WorldKernel` 和追加式世界账本仍是唯一事实源。任何新能力都不得绕过它写入生活史、日程、行动、经历或关系。

## 已完成的基底

| 能力 | 当前状态 | 说明 |
| --- | --- | --- |
| 追加式世界账本 | 已完成 | 世界事件是事实源；投影可删除、重建和哈希比对。 |
| 逻辑时间 | 已完成 | 时间推进、倍率与日程变化均记录为事件。 |
| 固定 NPC 与种子 | 已完成 | NPC、地点、可用时段、模板与长期目标来自 `world_seed.yaml`。 |
| 活动—结果—经历链 | 已完成 | 只有受规则验证的活动结果可形成可引用经历。 |
| 模型事实边界 | 已完成 | 模型候选必须附着于已完成活动，不能直接创建生活事实。 |
| 主动分享事务 | 已完成 | 选择、outbox、行动、投递与 `ExperienceShared` 可追踪。 |
| 外发不确定性 | 已完成 | 中断投递进入 `unknown`；没有回执不得盲目重发。 |
| 目标生命周期 | 已完成 | 支持延期、复查、恢复、放弃和补偿。 |
| 受限替代活动 | 已完成 | 只能使用种子允许的 `fallback_templates`。 |
| 休息作为事实 | 已完成 | 无可用活动可记录 `ActivityRested` 并恢复种子规定的精力。 |

## P0：启用真实聊天前必须补齐

### 1. QQ 平台投递回执对账

**问题**：内核支持用外部回执把 `unknown` 对账为 `delivered` 或 `failed`，但 QQ 适配器尚未明确提供可靠的回执查询路径。

**实施**：

- 为每次外发保留平台消息 ID、幂等键或可查询的投递凭据。
- 适配器启动恢复时，只查询 `unknown` 行动；取得证据后调用世界结算入口。
- 平台无法提供证据时保留 `unknown`，在面板中要求人工确认；禁止自动重发。

**验收**：发送后崩溃、重启、平台回执迟到三种情形下，不重复发消息、不把未确认消息写成已分享经历。

### 2. 世界模式旁路扫描

**问题**：旧运行时、旧生活表和社交任务仍保留作兼容回退；世界模式不能继续读取它们作为行为权威。

**实施**：

- CI 中扫描 `WORLD_RUNTIME_ENABLED` 路径内的 `life_runtime`、旧日程、旧社交任务写入。
- 每个外发消息断言能追到：命令、世界事件、行动、outbox、结算结果。
- dashboard 只允许提交命令，不允许直接修改投影数值。

**验收**：世界模式下任意生活经历、日程和主动消息均可回溯到 `world_events`。

### 3. 真实启用门禁

**实施**：

- 在目标数据库执行所有投影重建并要求哈希一致。
- 执行世界回放集、投递故障注入和重复消息幂等测试。
- 首次启用采用单用户、可关闭的 feature flag；出现异常只切回旧系统，不混写两个纪元。

**验收**：满足 [world-kernel.md](world-kernel.md) 的不变量，且没有未解释的 open / unknown 行动。

## P1：让世界产生更自然的有限选择

### 4. 目标优先级评分

**当前限制**：目标有效性与截止时间会限制活动，但多个可行活动之间仍主要由日程顺序决定。

**实施**：

- 在种子中为长期目标增加 `priority`、`urgency_window`、`minimum_progress` 和允许的替代活动。
- `LifeSimulation` 在同一时段的候选活动中计算确定性分数：截止临近度、剩余进度、精力成本、地点迁移、NPC 可用性和当日重复度。
- 选择过程追加 `ActivitySelected`，记录候选集合、拒绝理由、规则版本与最终分数；禁止随机或模型直接决定。

**验收**：同一事件流始终选出同一活动；临期目标可提高优先级，但不会突破精力、地点或边界限制。

### 5. 正式的延期与恢复日程

**实施**：

- 活动因资源不足、冲突、用户打断或 NPC 不可用时进入 `ActivityDeferred`，必须附带下一次复查时间和原因码。
- 到复查时间只能恢复、替代、取消或休息，不允许无限悬挂。
- 目标延期和活动延期使用不同事件；前者是长期承诺，后者是单次安排。

**验收**：跨日、跨周和离线跳跃后，所有未完成活动都有唯一状态和下一步策略。

### 6. 更完整但有限的活动模板网络

**实施**：

- 每个时段保留 2–3 个低风险模板；每个模板声明地点、时段、精力影响、目标贡献、NPC 权限和最大频率。
- 加入“独处整理”“家庭联络”“社团准备”“恢复性休息”等模板，不接入现实新闻、社交媒体或未注册人物。
- 对重复度设置日/周上限，避免每天发生同样的读书会或同样的社团互动。

**验收**：一周回放中，活动可有有限变化，但不会出现未注册 NPC、未定义地点或无来源事故。

### 7. 地点与过渡成本

**实施**：

- 用种子中的地点图和明确旅行时长，而不是接入真实地图。
- 计划或替代活动必须预留地点转换时间；冲突时选择延后、替代或休息。

**验收**：角色不会在同一时间段同时出现在上海、华师大和嘉兴。

## P1：让用户感受到连续性

### 8. 可分享经历的选择质量

**实施**：

- 主动分享评分只读取已提交经历：新鲜度、关系阶段、用户最近活跃度、主动频率、未读状态和重复度。
- 不分享也是合法结果；经历保留在世界中，用户追问时仍可引用。
- 用户发新消息取消尚未发送分享；已进入 `unknown` 的分享只可回执对账。

**验收**：一周内主动分享不刷屏、不重复；取消和失败不会被说成“已经告诉你”。

### 9. 用户追问的时间与经历检索

**实施**：

- 为“今天 / 昨天 / 某天 / 上次”建立逻辑时间范围查询。
- 回复只引用范围内已提交的经历、事实和已结算行动；无记录时明确保持不确定。

**验收**：日期追问、取消活动、未分享经历和失败投递都不会产生假叙事。

### 10. 世界—对话行为映射

**实施**：

- 将世界状态映射为少量可观察策略：忙碌、可回复、恢复中、主动性下降、目标临期。
- 策略只影响回复时机、长度、是否追问和是否创建行动；不能直接生成事实文本。

**验收**：同一用户消息在“刚休息”“活动中”“目标临期”三种状态下有可预测差异，且原因可在时间线查看。

## P2：验证、可观测性与运营

### 11. 7–14 天连续性回放集

覆盖：

- 忙碌、休息、延期、恢复和目标完成；
- 用户回来取消待发分享；
- 用户追问某日经历；
- 投递失败、未知投递回执、重复回调；
- NPC 不可用、资源不足、地点冲突；
- 世界暂停、倍率切换、长时间离线恢复。

**验收指标**：状态哈希确定、无重复经历、无未结算行动、无无来源引用。

### 12. 仪表盘与审计体验

**实施**：

- 展示逻辑时间线、活动选择理由、候选拒绝理由、目标进度、进行中行动与投递状态。
- `unknown` 显示为需要平台或人工对账的项目，不能伪装成失败或成功。
- 提供世界快照和投影重建报告。

### 13. 数据保留与世界纪元

**实施**：

- 新世界必须从新纪元开始；旧世界只读归档。
- 提供导出事件流、验证事件哈希和创建新纪元的受控工具。

## 暂不做的事项

以下能力在 P0/P1 未稳定前不建议增加：

- 新的情绪维度、人格标签或心理独白类别；
- 独立 NPC LLM、NPC 之间自由对话；
- 真实地图、新闻、社交媒体或传感器接入；
- 模型自由生成活动、人物、地点、冲突或“回忆”；
- 第二套记忆数据库或第二个世界状态机；
- 仅为“人味”而设计、却没有行动结算条件的状态。

## 决策原则

1. 先证明“发生过”，再优化“怎么说”。
2. 先做少量稳定选择，再扩展活动数量。
3. 不确定不是失败；无证据时保留不确定性。
4. 任何可观察线上行为都必须有开始、取消、失败、过期或结算路径。
5. 新机制若不能改变用户可见行为、不能回放、不能测试，就不应进入世界写模型。
