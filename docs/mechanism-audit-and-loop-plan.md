# 机制审计与闭环优化计划

更新日期：2026-07-11

## 结论

当前系统并非“没有机制”，而是已经有若干真实运行的机制，却缺少一个统一的**回合承诺**：
每个内部判断应明确决定什么可观察行为、何时完成、投递失败怎么办、下一轮如何被引用。

因此用户会看到模型的单句表达，却较少看到“我刚才这样做，所以现在自然会这样做”的连续因果。
接下来的目标不是继续扩充情绪和动作标签，而是把现有机制收敛为少数可回放的闭环。

## 审计范围与方法

本审计检查的是代码中的实际调用链与持久化结果，不以文档中存在某个概念作为“已接入”的证据。

闭环判定标准：

```text
输入/时间事件
  -> 可解释的状态变化
  -> 可观察的行为决定
  -> 成功或失败投递
  -> 可持久化的后续承诺/状态
  -> 下一轮可验证地消费或取消
```

只有完整经过这六步的机制才算“有效”；只改变 prompt、只写记忆、或只留下数值的机制为部分接入。

## 当前机制审计

| 机制 | 实际入口和出口 | 闭环状态 | 主要问题 |
| --- | --- | --- | --- |
| 情绪与关系 | `interpret_interaction` -> `transition_emotional_state` -> `MoodState` -> 上下文策略、生活投影、主动判断 | 部分有效 | 文本关键词分类过粗；多个情绪字段同时改变，但没有一个统一的“本回合关系行为”结果。 |
| 读消息/延迟回复 | `QQMessageCoalescer` -> `decide_phone_attention` / `decide_reply` -> `reply_later` 社交任务 -> 投递确认 | 有效 | 主要在 NapCat/WebSocket 合并器路径中体现；需要确认官方 QQ 等其他适配器是否走同一回合策略。 |
| 会话打断与分段 | 合并器、`conversation_pulse`、用户新消息取消任务、分段投递 | 有效 | 行为机制和情绪机制独立决策，容易出现“节奏正确但说的话不对应”。 |
| 未说出口/心理活动 | `infer_inner_subtext` 写入记忆并作为当前 prompt 倾向；`withheld_impulse` 在主动消息被拒后改变状态 | 部分有效 | 没有统一的、可结算的“未表达事务”；大部分结果只是提示词或泛化的 `unresolved_emotion`。 |
| 主动行为 | `proactive_tick`、冷却、未回应预算、投递反馈 | 部分有效 | 普通主动消息仍主要由模型决定内容和理由；应与未完成事务、关系评估共享一个决策结果。 |
| 虚构生活 | 日计划/生活运行时 -> 私有事件 -> 生活分享 | 有效（已收紧） | 已禁止模型凭空造生活事实；仍需把“生活事件”与“对话事务”在同一观测面中展示。 |
| 记忆检索 | 事实账本、相关记忆、上下文编排 | 部分有效 | 事实与检索记忆已经分层，但 LLM 生成的 consolidation/self-core 仍可能间接把叙事摘要变成角色解释。 |
| Self Core / Character-LLM 思路 | 后台合并达到阈值后 `build_self_core`，下轮作为 prompt block 注入 | 部分有效 | 它是摘要，不是行为权威；没有直接生成可验证的承诺、边界或待办。仓库中未发现 Hermes 的运行时适配器，当前只有文档中的未来模型替换提及。 |
| SillyTavern / EchoText | EchoText 思路已移植；SillyTavern 是可选 `conversation_core` | 部分有效 | 它们不是当前 daemon 的权威状态来源。启用 SillyTavern 时必须验证其输出仍经过同一事实、事务和投递闭环。 |

## 关键发现

### 1. 机制按“来源”组织，而非按“一个回合的结果”组织

当前一个用户消息可依次触发情绪状态、关系修复、生活投影、内在倾向、记忆写入、未答问题、表情建议、延迟决定等多个模块。它们各自合理，但调用方必须理解大量接口和顺序；这是一个浅的编排区，局部修改很容易旁路其他机制。

### 2. “心理活动”大多没有可观察的结算条件

“想说但忍住了”只有在之后满足下面之一时，才是用户能感到的心理：

- 形成一个可取消、可过期、可选择性补说的事务；
- 改变下一次回应的边界、长度、主动性或话题选择；
- 被后续用户行为明确消解。

如果只存一条描述性记忆，模型下一次未必会消费它，用户也无法感知其因果。

### 3. Self Core 的职责过宽

它同时概括身份、用户画像、关系、知识边界和未完成话题。它适合作为“摘要读模型”，不适合作为事实、情绪或事务的写模型。任何会影响行为的字段都应来自对应账本，而不是摘要模型的自然语言结论。

### 4. 线上动作的重点应是承诺，而非拟态细节

“看手机”“正在输入”“晚点回”是有效动作，因为它们改变时机并可被取消或完成。无来源的“我刚刚在做 X”不是动作机制，而是叙事。线上陪伴最有价值的动作是：看见、等待、回应、暂停、补回、放下、主动接近和守住边界。

## 目标闭环

建立一个深模块：`TurnCoordinator`。它是唯一决定一次线上互动如何落地的模块；适配器、状态机、模型和投递层都通过它协作，而不自行拼装回合行为。

其接口应保持很小：

```text
plan_incoming_turn(input) -> TurnPlan
confirm_delivery(turn_id, delivery_result) -> TurnCommit
advance_time(now) -> list[TurnPlan]
```

`TurnPlan` 不是自然语言，而是一个可审计的结构：

| 字段 | 含义 |
| --- | --- |
| `appraisal` | 本轮最重要的关系判断，例如“用户在示弱”“用户在收尾”“需要修复”。 |
| `attention_action` | 现在读、未读延迟、读到后延迟、立即回复、跳过。 |
| `expression_policy` | 回答/接住/设边界/收尾/轻分享；含问题预算和长度预算。 |
| `allowed_facts` | 本轮可使用的、带来源的事实。 |
| `open_commitments` | 新建、延期、取消或完成的对话事务。 |
| `observable_reason` | 面板和回放可读的简短理由，不能直接泄漏给用户。 |

模型只消费 `expression_policy + allowed_facts` 并返回候选文本；它不能创建事实、事务或状态。投递成功后才执行 `TurnCommit`。

## 分阶段优化计划

### Phase 0：观测与基线（先做）

目标：回答“某机制到底有没有改变用户看到的行为”。

1. 为每个入站、主动和延迟回复记录结构化 `turn_trace`：输入、状态差异、选中的事务、时机决定、允许事实、模型原文、清洗结果、投递结果。
2. 在调试面板按一条时间线展示 trace，而非只展示当前状态和 prompt。
3. 建立 25 个多轮回放场景，固定时间、随机种子和模型假实现；每个场景断言状态变化与用户可见结果。
4. 记录三个基线指标：
   - 事务完成率：创建后是否被完成、取消或过期；
   - 状态可见率：状态变化中有多少在两轮内改变了行为；
   - 无依据断言率：发送前被事实门拦截或被评测标记的比例。

验收：不依赖人工翻日志，可以解释任意一条消息“为什么此刻回复/不回复、为什么是这个语气”。

### Phase 1：收敛为回合承诺（P0）

目标：让心理、节奏、关系和投递在同一闭环中结算。

1. 引入 `TurnPlan`/`TurnCommit` 数据模型和持久化 trace，先作为现有 `CompanionEngine` 的内部 seam，不替换适配器。
2. 将 `reply_later`、`comfort_followup`、`promise_followup`、`conversation_pulse`、`life_share_followup` 统一为带 `origin_turn_id`、`reason_code`、`resolution` 的对话事务。
3. 将 `inner_subtext` 与 `withheld_impulse` 改为事务或短期 appraisal；没有结算条件的描述性记忆不再写入长期记忆。
4. 所有投递路径统一调用 `confirm_delivery` / `fail_delivery`，包括 SillyTavern、官方 QQ、NapCat 与调度器恢复路径。

验收：用户新消息、发送失败、超时、关系修复都能使一个事务走向唯一终态，不存在“写进去了但没人再消费”。

### Phase 2：减少状态，强化行为映射（P0）

目标：从“很多数字”变成“少数稳定选择”。

保留四个行为级评估：`security`、`respect`、`closeness/initiative`、`boundary`；保留情绪向量作为短期调制，但不再允许每个字段独立直接驱动 prompt。

为每个 appraisal 写一张映射表：

| appraisal | 当前可观察行为 | 后续变化 |
| --- | --- | --- |
| 用户脆弱 | 优先读、先接住、建立关心跟进 | 用户回来说好转则关闭；长期未回则自然降级。 |
| 被冒犯 | 不讨好、可读后延迟、短而清楚 | 道歉只部分恢复；多轮稳定互动才解除。 |
| 用户忙 | 不追发、取消普通余韵 | 到期后不自动索取回应。 |
| 温和回应 | 可稍增加展开度或轻分享 | 只影响后续一到数轮，不直接宣布亲密。 |

验收：同一句输入在不同 appraisal 下会产生可预测且可测试的不同“选择”，而不是只有形容词不同。

### Phase 3：把 Self Core 降为读模型（P1）

目标：让自我连续性服务于行为，而不成为第二个真相来源。

1. Self Core 只总结已验证事实、长期关系摘要和未完成事务 ID；禁止其生成新经历或独立关系判断。
2. 事实、世界事件、对话事务、关系评估分别保留各自的写模型；Self Core 只能读取。
3. 若尝试 Hermes 或其他 Character-LLM，实现为同一个 `SelfCoreBuilder` seam 的 adapter，输出必须通过来源校验并跑同一回放集。

验收：关闭 Self Core 后，事实、时机、边界和事务仍正确；开启后只提升摘要质量，不改变真相。

### Phase 4：表达质量与模型实验（P1）

目标：在闭环稳定后再优化“人味儿”。

1. 让模型输出受限结构：`reply_text`、`question_intent`、`mentioned_fact_ids`、`suggested_commitment`；daemon 验证后才采用。
2. 比较当前模型、SillyTavern 路径和候选 Hermes 模型在同一 25 个回放集上的表现。
3. 人工只评价“连续五轮是否像同一个人”，并记录具体失真类型；不要只评价单句文采。

## 首批回放场景

1. 用户连续三条铺垫故事：系统等待合并，但不会无限等待或误判为拒绝回应。
2. 用户说“我先忙”：她停止追发；用户回来后自然接上，而非抱怨或假装失忆。
3. 用户脆弱表达后沉默：先当下接住，后续关心只发一次且可被用户回来取消。
4. 用户冒犯后道歉：回复和接下来三轮的边界逐步变化，不瞬间归零。
5. 她有一个未说出口的主动冲动：用户先回来则取消；用户未回来且仍合适才发一次；两种情况都可追踪。
6. 投递失败：不把未发送文本写成聊天历史，也不机械重放。
7. 事实问句与日期问句：只能读取授权账本；无记录时保持不确定。
8. 同一情绪词但不同关系阶段：行为不同，不能仅替换撒娇程度。

## 暂停事项

在 Phase 0--2 完成前，暂停新增：情绪维度、人格标签、主动触发器、生活活动模板、内心独白类别，以及新的第三方记忆/角色框架。它们会扩大输入面，却不能解决当前“状态没有结算”的问题。
