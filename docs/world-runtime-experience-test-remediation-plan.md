### 0.4 新发现及迭代修复

1. 模型即使看到当前场景，也可能继续沿角色卡补写“赖床/课表”。修复方式不是放宽门禁，而是把
   当前场景做成只在当前 revision 有效的临时 source；时间推进后旧 source 自动失效。
2. 模型会填写 `claims.source_id` 却漏掉 `mentioned_event_ids`，或反过来。系统现在可从任一合法
   引用提示中重建逐字来源回复，不再需要把整轮丢弃。
3. 问“现在”时模型可能同时引用旧经历。安全降级现在按问题作用域裁剪：当前问题只用 scene，
   上午/下午/昨天问题只用对应逻辑时间范围内的经历。
4. 严格门禁会拒绝格式正确但内容无依据的回复。系统最多进行一次独立、已落账的修复模型调用；
   仍失败才使用无事实的短回复，避免模型调用循环。
5. `ready` 不能仅等于投影哈希一致。门禁现额外检查 completed 活动缺少 outcome、同一消息存在多个
   延迟 Action、开放/unknown Action 与投递回执策略。

- Self Core 不再是模型自由写入的散文，而是种子、目标、关系和已提交世界状态的确定性投影。

## 1. 结论与目标

本次使用最新代码、独立 SQLite 数据库和真实模型完成了从 2026-07-11 09:00 到
2026-07-19 20:30 的体验测试。事件账本、活动结果、NPC 互动、长期目标和投影重建可以
运行，但世界状态尚未可靠地传递到聊天行为；时间、资源和延迟回复也没有形成完整的运行闭环。

本计划的目标不是增加更多拟人机制，而是让以下链路在真实适配器入口上成立：

```text
逻辑时间推进
→ 按时间顺序结算生活活动
→ 形成有来源的事实与经历
→ 以稳定标识提供给回复模型
→ 校验并生成用户可见回复或合法的暂缓动作
→ 经适配器幂等投递并结算
→ 事件流可确定性重建
```

完成本计划后，“她做了什么”“她现在为什么没回复”“她说的经历来自哪里”都必须能从
同一个世界事件流解释，且不能因为模型输出格式错误退化成无意义回复或产生假历史。

## 3. 修复后的模块责任

不新增第二事实源。继续由 `WorldKernel` 作为唯一世界写模型，但收紧三个深模块的接口，
避免调用方重复解释世界状态。

### 3.1 `WorldClockDriver`

建议接口：

```text
tick(world_id, observed_now, expected_revision) -> WorldDecision
advance(world_id, target_logical_at, expected_revision) -> WorldDecision
```

模块内部负责：

- 根据暂停、实时、1×–8× 和上次观测墙钟计算目标逻辑时间；
- 将墙钟读数作为外部输入记录；
- 把长跳跃拆成有序结算点；
- 幂等处理同一观测时间；
- 绝不直接修改时钟投影。

daemon、面板和测试只跨这个 seam，不各自实现倍率换算。

### 3.2 `LifeSimulation`

保留单一推进接口：

```text
advance(world_id, target_logical_at, expected_revision) -> WorldDecision
```

模块内部必须按结算点顺序执行：

```text
恢复/睡眠结束
→ 活动到期结算
→ 需求与地点变化
→ 目标截止检查
→ 下一活动候选与可行性校验
→ 开始或延期/休息
```

不允许先将整周活动全部标为完成，再统一生成少量结果。任何
`ActivityStarted` 都必须已经通过精力、地点、旅行时间、NPC 可用性和模板资格校验。

### 3.3 `TurnCoordinator`

建议唯一对话执行接口：

```text
observe(message) -> TurnDisposition
resume(action_id) -> TurnDisposition
settle_delivery(action_id, result) -> WorldDecision
```

`TurnDisposition` 必须显式区分：

```text
ReplyScheduled(action_id)
Deferred(action_id, due_at)
Suppressed(reason)
Duplicate(original_receipt)
Rejected(reason)
```

路由、QQ 官方、NapCat、OneBot 和 simulator 只把 disposition 转成各平台可理解的响应。
合法沉默不再使用 `None` 表达，恢复动作也不再伪装成新的入站消息。

## 4. 分阶段实施计划

### Phase 0：冻结启用并建立失败测试

目标：先把本次真实故障固化为自动化测试，避免在修复过程中丢失证据。

工作项：

1. 将真实体验剧本转为独立世界数据库测试夹具。
2. 为 WR-P0-01 至 WR-P0-15 分别添加失败用例。
3. 临时收紧 `ready`：任何 P0 门禁未执行、投递回执不受支持或存在开放异常动作时返回
   `ready=false`。
4. 明确旧文档中“已完成”表示代码存在还是端到端通过；未经真实入口验证不得标记完成。

退出标准：15 项问题均能由测试稳定复现，且现有真实聊天启用门禁不会误报 ready。

### Phase 1：修复事实包与回复门禁

目标：让正确经历可以被引用，无依据陈述不能混入回复。

工作项：

1. `conversation_context()` 输出稳定的结构化来源：

   ```json
   {
     "source_id": "outcome:2026-07-11:morning_study",
     "source_type": "experience",
     "content": "在图书馆和范予安核对了读书会的书单。",
     "occurred_at": "2026-07-11T12:00:00+08:00",
     "reference_state": "committed"
   }
   ```

2. 当前逻辑时间、当前地点、当前活动、活动可中断性作为状态约束单独提供，不能伪装成经历。
3. 模型只能引用提供过的 `source_id`；禁止 `event_1` 等临时别名。
4. 校验器按陈述逐条验证，不能因为整段以问号结尾而跳过事实检查。
5. 校验失败时只移除或重写无依据片段；不要把整段统一替换为“我在。”。
6. 无记录时允许明确表达“不确定/没有可引用记录”，但不得补造生活细节。

退出标准：

- 空世界问“在做什么”不会生成起床、课表或图书馆事实；
- “上午做了什么”“昨天做了什么”能引用正确经历并保留真实 source ID；
- 活动中问“现在忙吗”只能使用当前世界状态，不把旧经历说成刚发生；
- 未提交候选、取消活动和失败投递不可引用。

### Phase 2：修复时间与生活资源闭环

目标：倍率、跨夜、地点、精力和长跳跃共享同一套确定性规则。

工作项：

1. daemon 周期调用 `WorldClockDriver.tick()`；暂停模式不推进，其余模式按观测墙钟和倍率推进。
2. 在种子中明确睡眠/休息窗口、恢复量、最大精力和被打断策略。
3. 增加有限地点图和确定性旅行时长；需要跨地点时生成旅行或延后事件。
4. 活动选择前检查最低精力；不足时只能延期、替代为恢复活动或取消。
5. `no_eligible_template` 必须终止选择链，不能继续 Started/Completed。
6. 长跳跃按活动开始、结束、恢复、复查和目标截止时间排序结算。
7. 目标截止检查必须发生在截止点之前所有可结算活动之后。
8. 活动完成但没有合法结果时，明确产生无结果终态或补偿事件，不能留下“完成但生活史无解释”。

退出标准：

- 8× 下墙钟 5 秒对应约 40 秒逻辑时间，允许小范围调度误差；
- 跨夜按种子规则恢复精力；
- 精力为 0 不会开始消耗型活动；
- 不会在无旅行时间的情况下从华东师大瞬移到其他地点；
- 一次推进 7 天与逐结算点推进 7 天得到相同状态哈希和业务事件序列；
- 目标不会先因截止延期，再被截止前本应已结算的活动倒序推进。

### Phase 3：修复通信 Action 与适配器语义

目标：回复、沉默、延迟和恢复在所有适配器上只有一条事务链。

工作项：

1. 使用 `TurnDisposition` 替代 `CompanionReply | None`。
2. HTTP simulator 对 `Deferred` 返回明确的 202/结构化状态，不产生 500。
3. 合并延迟注意力和延迟回复的所有权：一个用户消息只能有一个权威延迟 Action；已读状态作为
   事件投影，不再创建第二个可执行待办。
4. `resume(action_id)` 读取原消息和原因果链，不再次提交 `UserMessageObserved`。
5. 到期、用户新消息、取消、过期和恢复成功必须通过同一个状态迁移表互斥结算。
6. scheduler 只 claim 世界 due actions；旧 `social_tasks` 不得与世界 Action 同时执行。
7. simulator 使用模拟投递 adapter，QQ ID 校验只存在于 QQ adapter 内部。

退出标准：

- 低精力活动中收到消息只创建一个延迟 Action；
- 到期只生成并投递一次回复；
- 新消息取消待回复后，恢复器不能再次发送；
- simulator、QQ 官方和 NapCat 对同一 disposition 的世界事件序列一致；
- 合法沉默不返回服务错误。

### Phase 4：端到端幂等与外部结果一致性

目标：平台重试、进程重启和重复回调不会产生第二次模型调用或第二条回复。

工作项：

1. 以 `platform + platform_user_id + message_id` 作为入站幂等键；没有平台 message ID 时使用受控
   内容指纹并记录其限制。
2. `world_command_receipts` 保存第一次处理的 disposition 与最终 action ID；重复请求直接返回原收据。
3. 模型调用 action 的 idempotency key 必须唯一对应一次已记录外部结果；禁止同一 action ID 追加不同提议。
4. outbox 与 `ActionScheduled` 同事务创建；重复执行只读取已有 outbox。
5. 投递 adapter 使用 world action id 作为平台侧可用的幂等标识；没有平台幂等能力时使用发送中
   `unknown` 与回执对账，禁止盲目重发。
6. 对并发相同消息、过期 revision、重复恢复、迟到回调建立故障注入测试。

退出标准：同一消息提交 100 次只产生一个 `UserMessageObserved`、一次模型外部结果、一个
outgoing Action 和最多一次用户可见投递。

### Phase 5：真实启用门禁与观察期

目标：`ready=true` 代表真实聊天链路可用，而不只是投影可以重建。

门禁必须同时检查：

1. 所有投影从空重建且哈希一致；
2. 没有无期限或互相重复的开放 Action；
3. 延迟回复恢复剧本通过；
4. 重复消息与重复外部回调剧本通过；
5. 7 天逐步推进与长跳跃等价；
6. 所有回复声明均能追到来源或被标记为非事实表达；
7. 当前启用的投递 adapter 支持项目所要求的回执/unknown 策略；
8. 数据库和运行代码的规则版本、种子版本兼容；
9. 不存在旧生活表、旧社交任务或旧 MoodState 写入旁路；
10. P0 测试报告写入审计记录，并包含测试代码版本和世界 revision。

启用顺序：

```text
isolated simulator
→ 本地 QQ sandbox
→ 单一测试 QQ 账号
→ 24 小时观察
→ 7 天观察
→ 正式账号
```

任何阶段出现重复投递、假经历、资源穿透或无法解释的开放 Action，都应将门禁恢复为
`ready=false`。不得通过同时写旧世界来回退。

## 5. 必须新增的验收剧本

### 5.1 事实与聊天

- 空世界：“你现在做什么？”
- 活动进行中：“你忙吗，现在在哪？”
- 当日回忆：“上午/下午做了什么？”
- 跨日回忆：“昨天做了什么？”
- 指定日期：“7 月 11 日你做了什么？”
- 取消活动追问、未提交候选追问、投递失败追问。
- 一条回复中混合事实、感受和提问，验证只检查事实声明而不误杀整段。

### 5.2 时间与生活

- 暂停、实时、1×、2×、4×、8×；
- 跨小时、跨日、跨周和进程离线恢复；
- 睡眠恢复、低精力延期、恢复性活动；
- 同地点连续活动、跨地点旅行、旅行时间不足；
- NPC 不可用、模板频率达到上限、无合格模板；
- 目标在截止前完成、截止时延期、复查后恢复或放弃。

### 5.3 通信与故障

- 活动中立即回复、低精力延迟、免打扰、合法沉默；
- 延迟到期、用户提前返回、用户撤销或发送新消息；
- 相同消息 ID 串行和并发重投；
- 模型返回后崩溃、outbox 创建后崩溃、投递调用中崩溃、回执迟到；
- scheduler 重启、重复 claim、重复外部回调；
- simulator、QQ 官方和 NapCat 使用同一套世界事务断言。

## 6. 完成定义

本计划只有同时满足以下条件才可标记完成：

- WR-P0-01 至 WR-P0-15 全部有回归测试且通过；
- 不存在绕过 `WorldKernel` 的世界状态或通信行为写入；
- 8× 模式可以真实推进世界，而不是只改变显示字段；
- 7 天连续推进和一次长跳跃的最终哈希与关键事件语义一致；
- 精力、地点、模板资格和 NPC 可用性在 ActivityStarted 前生效；
- 用户追问已提交经历时得到有内容且有真实来源 ID 的回复；
- 无依据内容不能通过问句、空 claims 或错误别名绕过门禁；
- 延迟回复不会返回 500，不重复、不自我取消，并具有唯一终态；
- 相同平台消息重复投递不会产生第二条用户可见回复；
- 真实启用门禁在任一关键测试失败时返回 `ready=false`；
- 连续 7 天测试世界中没有假经历、瞬移、资源穿透、重复投递和无法解释的开放 Action。

## 7. 明确不采用的修复方式

- 不降低事实门禁来换取“回复看起来正常”；
- 不用提示词要求模型猜测事件 ID；
- 不在 QQ、NapCat、HTTP 路由中分别实现延迟和幂等；
- 不增加第二个定时器状态库或第二套生活状态机；
- 不用旧 `social_tasks`、`life_runtime` 或 `MoodState` 暂时补洞；
- 不把休息、旅行和资源不足写成仅供展示的文字；
- P0 未通过前不继续增加情绪维度、活动模板或自由生成事件。

## 8. 文档状态修正原则

现有文档中的“已完成”需在实施本计划时按以下定义重新核对：

- **模块存在**：代码和事件类型已建立，但未证明完整链路；
- **模块测试通过**：模块接口测试通过；
- **适配器集成通过**：至少 simulator 与一个真实投递 adapter 通过；
- **端到端完成**：真实入口、故障恢复、幂等、投影重建和用户体验剧本全部通过。

只有最后一种可以写为“世界闭环已完成”。
