# 世界模式对话「人味」修复闭环

更新时间：2026-07-12
状态：Iteration 9 完成，等待提交
范围：世界模式回复的言语行为响应、共情表达、人格与关系口径
不在范围：小屋/room、自由 NPC、世界事实账本重构、单纯堆叠文风形容词

## 1. 目标

事实正确只是底线。回复还必须先接住用户这一刻在做什么——倾诉、吐槽、纠正、求陪伴、追问或试探关系——再决定是否引用旧事实。沈知栀的表达应保持短、具体、慢热、有判断，不因模型发挥或安全降级突然变成客服、心理咨询模板或越级恋人。

```text
用户当前言语行为
→ 世界状态与关系调制
→ 先回应当下，再按需引用事实
→ 事实 grounded audit
→ 文风/共情/关系门禁
→ 一次定向修复
→ 同样受门禁约束的确定性降级
```

## 3. 第一轮公共测试缝隙

只在 `CompanionEngine.handle_message` 可观察边界写红灯；模型使用可控外部替身，世界内核与候选校验保持真实。纯判定规则可另写少量表驱动测试，但不能代替端到端回复测试。

### 红灯门禁

1. 用户明确说“别劝，陪我吐槽”时，建议式候选必须进入 repair；二次仍错时，降级必须共同站在用户这一边，不能继续建议或只反问。
2. 用户表达“做这么久最后还是没人味”的担心时，回复不得把旧的胃、咖啡、睡眠事实拉来抢话题；失败降级要复述当前担心的核心，而非泛化安慰。
3. 当前仍是慢热/刚认识关系时，“宝宝、永远爱你、只属于你”等越级口径必须拒绝并修复；二次失败要明确慢慢认识，而不是暧昧升级。
4. repair prompt 必须明确：先完成当前言语行为，再引用相关事实；不为显得共情而编造共同经历、心理或环境；保持一两句手机私聊和角色关系边界。
5. 所有修复后的候选仍通过既有 claim validator 与独立 grounded audit。

## 4. 实施计划

### Iteration 1

1. 建立上述三个端到端红灯；
2. 新增小而保守的 human-reply contract，只拦高置信错位：违背“别劝”、当前脆弱话题被旧健康事实劫持、关系越级、说明书式列点；
3. 在首次候选和 repair 候选上使用同一门禁，且门禁位于 grounded validation 之后、独立 audit 之前；
4. 为吐槽、当前担心和关系试探补充角色一致的确定性降级；
5. 更新主生成与 repair 指令，强调“先当下、后事实”；
6. 跑专项、世界/引擎回归与 lint，记录红绿结果。

## 5. 完成门禁

- 三类红灯全部转绿；
- 既有世界事实、主体、Action、并发专项无回归；
- grounded audit 调用与失败闭合路径不被绕过；
- 新规则不把普通建议、正常事实追问或逐步变暖的表达一概封死；
- 下一轮真实 30 回合回放需逐条标注：`speech_act / empathy / persona / grounding`，任一硬失败即回写本文件继续迭代。
