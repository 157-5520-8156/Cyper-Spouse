# 视觉小屋延期方案

更新日期：2026-07-10

本文记录“知栀的小屋”从当前演示面板升级为真正虚拟家园的路线。当前结论：
这条路线值得保留，但暂缓执行。短期工程重点回到状态机、生活运行时、记忆与
对话闭环，因为这些直接决定她是否像真人。

## 当前版本定位

当前 dashboard 的小屋是生活状态的视觉投影：

- 背景是一张完整等距像素图。
- 知栀是独立 sprite 图层。
- daemon 输出 `scene.location/action/expression/phone_attention`。
- 前端按锚点移动和切换姿态。

它适合做“观察当前状态”的面板，但不是完整游戏场景。它不能作为事实来源，也不能
反向修改生活运行时。

## 当前问题

| 问题 | 原因 | 影响 |
| --- | --- | --- |
| 人物会穿墙或穿家具 | 背景是整张插画，前端不知道墙、床、桌子在哪里 | 移动时破坏真实感 |
| 遮挡关系不稳定 | 家具没有前景遮挡层和深度排序 | 人物站位会显得贴在图上 |
| 动作库不足 | 只有站立、走路、看手机、睡觉等粗动作 | “洗漱、窝着看视频、整理东西”等生活动作无法演出来 |
| 场景不可扩展 | 小屋逻辑和背景绑定，学校/街边/咖啡店没有统一格式 | 后续每个场景都会变成重新手写 |
| 美术一致性风险 | AI 插画级背景和拆分 sprite 难完全一致 | 实体化初版可能比当前整图更丑 |

## 推荐长期路线：混合实体化

不要直接把整张图推倒重做成纯 tilemap。更现实的路线是混合实体化：

```text
AI 美术母版
  + 地板/墙体底层
  + 家具前景遮挡切片
  + 碰撞多边形
  + 交互锚点
  + 角色动作 sprite
  + 光照/氛围层
```

这样可以尽量保留当前漂亮的整体美术，同时解决穿墙、遮挡和动作锚点问题。

## 目标场景格式

未来每个场景应由数据描述，而不是写死在 `dashboard_ui.py`：

```json
{
  "id": "home_evening",
  "background": "/assets/scenes/home/background.png",
  "overlay": "/assets/scenes/home/foreground.png",
  "walkable": ["polygon refs or tile ids"],
  "colliders": ["wall", "bed", "desk", "sofa"],
  "anchors": {
    "desk.study": {"x": 264, "y": 673, "pose": "sit_laptop"},
    "sofa.watch_video": {"x": 544, "y": 858, "pose": "curl_phone"},
    "vanity.wash": {"x": 1044, "y": 783, "pose": "wash_face"},
    "bed.sleep": {"x": 1119, "y": 683, "pose": "sleep_side"}
  },
  "depth_sort": ["background", "actor", "foreground"],
  "lighting": "warm_evening"
}
```

核心原则：

- 场景数据负责“哪里能走、哪里会挡住人”。
- daemon 只说她在做什么，不负责像素坐标。
- 前端负责把状态映射成场景、锚点、动作和层级。
- 未发生的计划不能因为显示在场景里而变成生活事实。

## 动作库目标

先做高频生活动作，不追求全覆盖：

| 动作 | 对应生活状态 | 最低表现 |
| --- | --- | --- |
| `idle` | 空档、发呆 | 站立轻微呼吸 |
| `walk` | 场景移动 | 四方向走路 |
| `study_laptop` | 自习、上课间隙 | 坐在桌前、屏幕光 |
| `read_phone` | 看消息 | 拿手机低头 |
| `type_phone` | 正在组织回复 | 手机打字小动效 |
| `ignore_phone` | 收到但不看 | 手机亮一下，人不动或背对 |
| `curl_video` | 窝着看视频 | 沙发蜷坐、屏幕光 |
| `eat_simple` | 吃饭/喝水 | 坐在桌边、杯子/碗 |
| `wash_face` | 洗漱 | 站在洗手台前 |
| `tidy` | 收拾东西 | 小范围来回动作 |
| `sleep_side` | 睡觉 | 床上侧躺 |
| `pout_idle` | 别扭/被冒犯 | 背对或手机放下 |

## 扩展场景

等小屋实体化稳定后，再考虑：

- `campus_classroom`：上课、自习、课间看手机。
- `library`：安静学习、收到消息但不方便回。
- `campus_walk`：去食堂/教学楼路上。
- `cafe`：外出、和同学聊天、随手拍。
- `street_evening`：散步、买东西、低概率主动分享。

扩展场景前必须先完成统一场景 schema，否则会重复造轮子。

## 暂缓原因

当前用户最在意的是：

```text
她是否像一个真实存在的人
  > 她的小屋是否像完整游戏
```

视觉实体化会消耗大量资产制作、动作绘制、碰撞标注和调试时间。它能提升面板观感，
但不能直接解决以下核心问题：

- 她是否会根据用户行为产生长期印象变化；
- 她是否会有自然的未读、已读、犹豫、补回、主动发散；
- 她是否能避免幻觉、复读、误归因；
- 她的日常生活是否与记忆、情绪、主动消息闭环；
- 跨 QQ 小号、官方机器人、未来微信是否共享同一人格状态。

因此短期暂停视觉实体化，回到动态闭环与状态机完善。

## 恢复条件

满足以下条件后再启动：

1. 状态机与生活运行时闭环稳定，至少有一套持续对话评测集。
2. 面板需要观察的问题已经从“状态是否正确”转向“生活是否可视化”。
3. 确定愿意投入资产制作时间，或找到可合法复用的高质量等距室内资产。
4. 先做一个场景 schema 原型，不直接在当前 dashboard 上堆逻辑。

## 下一步不做什么

- 不继续精修当前假底图的碰撞。
- 不给每个家具手写一次特殊逻辑。
- 不为了小屋牺牲状态机、记忆和消息时机。
- 不把视觉状态当成生活事实写入数据库。
